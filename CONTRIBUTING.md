## 如何贡献

### 1. Fork 仓库

点击页面右上角的 "Fork" 按钮，将仓库复制到您的账户下。

### 2. 克隆仓库

```bash
git clone https://github.com/你的用户名/CDDA.git
cd CDDA
```

### 3. 创建分支

```bash
git checkout -b feature/your-feature-name
# 或
git checkout -b fix/your-bug-fix
# 或
git checkout -b translation/document-name
```

分支命名规范：
- `feature/` - 新功能
- `fix/` - 错误修复
- `translation/` - 文档翻译
- `docs/` - 文档改进
- `refactor/` - 代码重构

### 4. 进行更改

根据您的贡献类型进行相应的更改（见下文详细说明）。

### 5. 提交更改

```bash
git add .
git commit -m "类型: 简短描述

详细描述（可选）"
```

### 6. 推送到 GitHub

```bash
git push origin feature/your-feature-name
```

### 7. 创建 Pull Request

1. 访问您 Fork 的仓库页面
2. 点击 "Pull Request" 按钮
3. 填写 PR 标题和描述
4. 提交 PR

## 翻译文档

### 准备工作

1. **阅读翻译指南**：[CDDA文档翻译指南.md](doc/CDDA文档翻译指南.md)
2. **查看翻译进度**：[翻译进度.md](doc/翻译进度.md)
3. **选择文档**：选择一个未翻译的文档

### 翻译流程

#### 步骤 1：声明翻译任务

在开始翻译前，请在 [Issues](https://github.com/IoIqq/CDDA/issues) 中创建一个任务：

```
标题：[翻译] 文档名称.md
内容：
- 文档路径：doc/原版/2026.6.1/文档名称.md
- 预计完成时间：X天
- 翻译工具：（如使用 AI 工具请注明）
```

#### 步骤 2：进行翻译

1. 从 `doc/原版/2026.6.1/` 复制原文件
2. 按照翻译指南进行翻译
3. 保存到 `doc/翻译/` 目录（保持相同的目录结构）

#### 步骤 3：质量检查

- [ ] 格式是否保持一致
- [ ] 代码块是否正确
- [ ] 链接是否有效
- [ ] 术语是否统一
- [ ] 标点符号是否正确
- [ ] 使用 Markdown 预览工具检查渲染效果

#### 步骤 4：提交 PR

PR 标题格式：`翻译: 文档名称.md`

PR 描述模板：
```markdown
## 翻译信息

- 文档名称：文档名称.md
- 原文路径：doc/原版/2026.6.1/文档名称.md
- 译文路径：doc/翻译/文档名称.md
- 文档行数：XXX 行
- 翻译工具：（如使用）

## 翻译说明

- 遵循翻译指南的所有规则
- 术语翻译参考已有文档
- 代码块添加了中文注释

## 检查清单

- [x] 格式保持一致
- [x] 代码块正确
- [x] 链接有效
- [x] 术语统一
- [x] 已预览渲染效果
```

### 翻译规范要点

1. **标题格式**：`# English Title（中文标题）`
2. **保持格式**：Markdown 格式、代码块、缩进完全不变
3. **代码不译**：代码保持英文，仅添加中文注释
4. **术语统一**：使用统一的专业术语

详细规范请查看：[CDDA文档翻译指南.md](doc/CDDA文档翻译指南.md)

## 提交模组

### 模组要求

1. **完整性**：包含 `modinfo.json` 和所有必要文件
2. **文档**：提供 README.md 说明模组功能
3. **测试**：确保模组在最新版本 CDDA 中可用
4. **兼容性**：说明与其他模组的兼容情况

### 提交流程

1. 在 `mod/` 目录下创建模组文件夹
2. 添加模组文件和 README.md
3. 在主 README.md 中添加模组说明
4. 提交 PR

PR 标题格式：`模组: 模组名称`

## 报告问题

### 问题类型

- 🐛 **Bug 报告**：翻译错误、格式问题
- 💡 **功能建议**：新功能或改进建议
- 📝 **文档问题**：文档不清晰或错误
- ❓ **问题咨询**：使用问题或疑问

### 报告模板

#### Bug 报告

```markdown
**问题描述**
简要描述问题

**位置**
- 文件：doc/翻译/文件名.md
- 行号：第 XX 行

**期望行为**
描述期望的正确内容

**实际行为**
描述当前的错误内容

**建议修改**
（可选）提供修改建议
```

#### 功能建议

```markdown
**功能描述**
描述建议的功能

**使用场景**
说明为什么需要这个功能

**实现方案**
（可选）提供实现思路
```

## 代码规范

### Python 代码

- 使用 PEP 8 代码风格
- 添加必要的注释和文档字符串
- 函数和变量使用描述性名称

### JavaScript 代码

- 使用 ES6+ 语法
- 使用 2 空格缩进
- 添加必要的注释

### JSON 文件

- 使用 2 空格缩进
- 保持键的顺序一致
- 添加注释说明（使用 `//` 或 `"//"`）

## 提交规范

### 提交信息格式

```
类型: 简短描述（不超过 50 字符）

详细描述（可选，72 字符换行）

相关 Issue: #123
```

### 提交类型

- `翻译` - 文档翻译
- `修复` - 错误修复
- `功能` - 新功能
- `文档` - 文档改进
- `重构` - 代码重构
- `测试` - 测试相关
- `构建` - 构建系统或依赖更新

### 示例

```bash
# 好的提交信息
git commit -m "翻译: 完成 MONSTERS.md 文档翻译"
git commit -m "修复: 修正 MODDING.md 中的格式错误"
git commit -m "功能: 添加批量翻译脚本"

# 不好的提交信息
git commit -m "更新"
git commit -m "修改了一些文件"
git commit -m "fix bug"
```

## 审核流程

### PR 审核标准

1. **内容正确性**：翻译准确，无明显错误
2. **格式规范**：遵循项目规范
3. **代码质量**：代码清晰，有必要注释
4. **测试通过**：相关功能正常工作

### 审核时间

- 通常在 1-3 天内完成审核
- 复杂的 PR 可能需要更长时间
- 如有问题会在 PR 中留言讨论

### 合并条件

- ✅ 至少一位维护者批准
- ✅ 所有讨论已解决
- ✅ CI 检查通过（如有）
- ✅ 无冲突
